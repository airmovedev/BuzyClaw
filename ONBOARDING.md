> 核心原则：小白 3 步完成设置，老用户 1 步迁移完成

### 2.4 迁移数据清单

| 数据 | 源路径 | 目标路径 | 处理方式 |
|------|--------|---------|---------|
| 运行配置 | `~/.openclaw/config.yaml` | `~/Library/Application Support/ClawTower/config.yaml` | 读取并转换（保留 model、API 配置；覆盖端口、路径） |
| 工作区 | `~/.openclaw/workspace/` | `.../ClawTower/workspace/` | 整目录拷贝（MEMORY.md、SOUL.md、AGENTS.md、USER.md、IDENTITY.md、second-brain/ 等） |
| Agent 数据 | `~/.openclaw/agents/` | `.../ClawTower/agents/` | 整目录拷贝（会话历史、transcript） |
| 任务 | `~/.openclaw/tasks.json` | `.../ClawTower/tasks.json` | 直接拷贝 |
| 项目 | `~/.openclaw/Projects/` | `.../ClawTower/Projects/` | 整目录拷贝 |
| API 凭证 | `config.yaml` 中的 key/token | macOS Keychain | 提取后存入 Keychain，不落盘明文 |

### 2.5 迁移规则

1. **只拷贝，不移动** — 原始 `~/.openclaw/` 数据保持不动，用户可随时回退到 CLI 版
2. **config.yaml 需要转换** — 保留用户配置（model、API key、agent 设置、cron jobs），覆盖 ClawTower 专属配置（端口、数据目录、日志路径）
3. **冲突检测** — 如果检测到 OpenClaw Gateway 正在运行（检查进程或端口占用），提示用户先关闭 CLI 版再迁移
4. **API 凭证迁移** — 从 config.yaml 提取 API key / OAuth token，存入 macOS Keychain，ClawTower 配置中改为引用 Keychain
5. **幂等性** — 多次导入不会重复创建数据，已存在的文件跳过或覆盖（用户可选）

### 2.6 迁移后的身份继承

迁移完成后，ClawTower 自动继承：
- Agent 名字和 emoji（从 IDENTITY.md 读取）
- 用户昵称和偏好（从 USER.md 读取）
- Agent 性格和记忆（SOUL.md、MEMORY.md）
- 所有对话历史和第二大脑内容

**用户感知：** 打开 ClawTower，一切跟之前一样，只是换了个更好看的壳。

## 3. 新用户各步骤详细设计

### Step ⓪ 环境检测（自动）

App 启动时静默检测：

| 检测项 | 失败处理 |
|--------|---------|
| `~/.openclaw/` 是否存在 | 决定走新用户/老用户路径 |
| 网络连接 | ⚠️ "需要网络连接来设置 AI 账号" + 重试 |
| iCloud 登录状态 | ⚠️ "登录 iCloud 可解锁 iPhone 远程控制"（可跳过） |
| macOS 版本 | ❌ 低于最低要求则提示升级系统 |
| 磁盘空间 | ⚠️ 低于 500MB 提示清理 |

### Step ① 欢迎页

**视觉：**
- 全屏沉浸式，深色/浅色跟随系统
- 中央大 icon + 产品名
- 下方 3 个卖点卡片轮播：
  - 🤖 "AI 助手，私密运行在你的 Mac 上"
  - 🔒 "数据不上传，只属于你"
  - 📱 "iPhone 远程控制，随时随地"

**文案原则：**
- 不提 OpenClaw、Node.js、Gateway 等技术名词
- 强调「私密」「本地」「你的」

**操作：** 一个按钮 → [开始设置]

**OAuth 流程：**

2. 系统浏览器打开 OAuth 授权页

- 展开输入框：Provider 选择 + API Key 输入
- 输入后自动验证（调一个简单 API 测试）
- 成功 → ✅ → 下一步

### Step ⑤ 初始化

**后台操作（对用户不可见）：**
1. 从 App Bundle 解压 OpenClaw Runtime 到数据目录（首次）
2. 生成 `config.yaml`（基于用户选择的 Provider + Token）
3. 生成 `IDENTITY.md`、`USER.md`、`SOUL.md`（基于 Step ③ 信息）
4. 启动 Node.js 子进程运行 Gateway
5. 等待 Gateway `/health` 返回 200
6. 发送首条系统消息触发 Agent 初始化

**用户看到的：**
- 柔和的加载动画（脉冲呼吸灯 or 粒子效果）
- 分阶段提示文字：
  - "正在准备你的 AI 助手..."
  - "快好了..."
  - "✅ 准备好了！"
- 整个过程 5-10 秒

**失败处理：**
- Gateway 启动超时（30s）→ "出了点小问题，正在重试..." → 自动重启一次
- 连续失败 → "启动失败了，可能需要重启 App" + 发送诊断日志按钮

## 8. 关键交互细节

### 8.1 Agent 状态指示

用户不需要知道 Gateway 是什么，只需要知道助手状态：

| 底层状态 | 用户看到的 | 说明 |
|---------|-----------|------|
| Gateway 运行中 | 🟢 "在线" | 正常 |
| Gateway 启动中 | 🟡 "正在启动..." | App 刚打开 |
| Gateway 崩溃 | 🔴 "离线 · 正在恢复" | 自动重启（指数退避，最多 5 次） |
| API Token 过期 | 🟠 "需要重新登录" | 弹窗引导 |
| 无网络 | ⚪ "无网络连接" | 系统级提示 |

### 8.2 错误处理原则

- **不展示技术细节**（不说 "Gateway port 18789 connection refused"）
- **说人话**（"AI 助手暂时无法响应，正在自动恢复"）
- **自动恢复优先**（指数退避重试，都失败再提示用户）
- **一键反馈**（出错时提供"发送诊断报告"按钮，日志自动脱敏）

### 8.3 后台运行

- macOS：关闭窗口 ≠ 退出，Gateway 继续后台运行（Menu Bar 常驻）
- Menu Bar icon：显示状态 + 快捷操作（打开对话 / 暂停 / 退出）
- 完全退出：Menu Bar → 退出，同时 kill Gateway 子进程
